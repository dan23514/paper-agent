## 1. 概要

### 1.1 目的
ユーザーから提供される研究要素 (テーマ・手法・データ・結果・解釈) を入力として、Q1/Q2 ジャーナル掲載論文のみを引用した日本語の学術論文ドラフトを生成し、ファクトチェック・マルチAIレビューを経て Word (.docx) 形式で出力する半自動エージェントを構築する。

### 1.2 設計原則
1. **Claude Code 単一環境完結**: 追加のフレームワーク (LangGraph 等) を導入せず、Claude Code のスラッシュコマンド・Subagent・Skill・MCP で構成する。
2. **Human-in-the-loop によるコスト圧縮**: ChatGPT・Gemini のレビューはユーザーが Web 版を手動で呼び出すリレー方式とし、API 従量課金を回避する。Claude レビューは Claude Code 内で自動実行する。
3. **引用ファクトチェックの独立化**: 汎用 LLM の引用捏造リスクを排除するため、ファクトチェックは NotebookLM に分離する。
4. **チェックポイント駆動**: 全自動ではなく、明示的なチェックポイントごとに成果物を出力し、ユーザーが介入・承認してから次工程に進む。
5. **分野設定可能性**: 初期実装は理学・工学のみ。将来的に医学・社会科学等を追加可能な設定駆動構造とする。

### 1.3 スコープ外
- 図表の生成 (ユーザーが作成し、最終工程で挿入する)
- 査読対応 (リバイズ作業) — 本エージェントは初稿ドラフトの作成までを担う
- 投稿先ジャーナル管理 (Cover Letter 生成等)

### 1.4 想定コスト
- Claude Code Pro または Max のサブスクリプション
- OpenAlex API: 完全無料（登録不要）
- Exa MCP: 補完用途のみのため無料枠内に収まる想定
- CrossRef API: 完全無料
- NotebookLM, ChatGPT (web), Gemini (web): いずれも無料枠を活用

## 3. 動作フロー (チェックポイント設計)

### 3.1 全体フロー

```
CP0  入力受領                    ── /load
  ↓
CP1  文献検索 + Q1/Q2フィルタ   ── /search
  ↓ (ユーザー確認: 引用候補リスト)
CP2  アウトライン生成            ── /outline
  ↓ (ユーザー確認: 構成承認)
CP3  ドラフト執筆                ── /draft
     (本文 + Abstract + Keywords、引用 ID 検証付き)
  ↓
CP4  マルチAIレビュー            ── /review-packet
     (Claude: 自動 / ChatGPT・Gemini: Human-in-the-loop)
  ↓ (ユーザー: 外部 AI にリレー → 結果貼戻し)
CP5  Factcheck                  ── /factcheck-packet
  ↓ (ユーザー: NotebookLM にリレー → 結果貼戻し)
CP6  統合・修正                  ── /integrate
  ↓ (ユーザー確認: 最終承認)
CP7  Word 出力                  ── /export
```

### 3.2 チェックポイント詳細

#### CP0: 入力受領
- 入力: 後述「§4 入力仕様」のファイル
- 出力: 構造化された研究要素 (内部 YAML `cp0_input.yaml`)、検出された欠損項目の警告

#### CP1: 文献検索 + Q1/Q2 フィルタ
- 入力: テーマ、手法、結果のキーワード
- 処理:
  1. OpenAlex API で関連論文を主検索 (目標 40〜80 件)
  2. ヒット数が閾値未満の場合、Exa MCP で補完検索
  3. DOI が欠損している論文は CrossRef API で補完
  4. 各論文の ISSN から SJR データで Quartile を判定
  5. Q1/Q2 のみを残し、年代・関連性・被引用数で再ランク
  6. 上位 N 件 (デフォルト 20) を候補リスト化
  7. `bib_fetcher.py` で全候補の BibTeX を `cp1_references.bib` に出力
- 出力:
  - `references_candidates.md` (DOI, 著者, 誌名, Quartile, 関連性スコア)
  - `cp1_references.bib` (BibTeX)
  - `missing_bibtex.md` (BibTeX 取得失敗論文のリスト)
- ユーザー確認: `references_candidates.md` を編集して採用/除外を決定

#### CP2: アウトライン生成
- 入力: 研究要素 + 承認済み引用候補
- 処理: 理学・工学標準の IMRaD 構成 (Introduction / Methods / Results / Discussion / Conclusion) を生成。各セクション見出しと 200 字程度の要約を含む
- 出力: `outline.md`
- ユーザー確認: 編集可能

#### CP3: ドラフト執筆
- 入力: アウトライン、研究要素、引用リスト (`references_candidates.md`)
- 処理:
  1. Writer Subagent が IMRaD 各セクションを執筆
  2. 引用は `references_candidates.md` の ID 列の値 (例: `[REF: R1]`, `[REF: R2]`) のみを使用。**リスト外の ID を生成することを禁止する**
  3. Abstract (目標字数: `abstract_chars` に従う) を生成
  4. Keywords (5〜8 語) を生成
  5. **引用 ID 検証ステップ**: 出力テキスト中の `[REF: *]` を全抽出し、`references_candidates.md` の ID と突合。存在しない ID が 1 件でもあればセクションを再生成 (最大 2 回リトライ)
  6. 完了したセクションを `progress.yaml` に記録（§10.5 参照）
- 出力: `draft_v1.md` (Abstract・Keywords・本文、引用は検証済み仮 ID)

#### CP4: マルチ AI レビュー
- 処理:
  - **Claude レビュー (自動)**: `/review-packet` 実行と同時に reviewer Subagent が `draft_v1.md` を評価し `results/claude.md` を自動生成
  - **ChatGPT・Gemini レビュー (Human-in-the-loop)**: 以下のパケットを出力
- 出力: `review_packet/` フォルダ
  - `draft_v1.md` (本文)
  - `prompts/chatgpt.md`, `prompts/gemini.md` (各 AI 用に最適化したレビュー指示)
  - `results/claude.md` (自動生成済み)
  - `results/chatgpt.md`, `results/gemini.md` (空。ユーザーが貼り付ける)
  - `instructions.md` (ユーザー向けマニュアル)
- ユーザー作業:
  1. Claude レビュー結果 (`results/claude.md`) を確認
  2. ChatGPT・Gemini にプロンプトを投げ、結果を各 `results/` ファイルに保存
  3. Claude Code に戻り `/integrate-review` で次工程へ

#### CP5: Factcheck (Human-in-the-loop)
- 出力: `factcheck_packet/`
  - `claims.md` (本文中の全引用について「主張 + 引用論文 + DOI」のテーブル)
  - `pdfs/` (OpenAlex/arXiv/PubMed Central 等から自動 DL した OA-PDF)
  - `missing_pdfs.md` (ユーザーが手動で追加すべき非 OA-PDF リスト)
  - `notebooklm_prompts.md` (NotebookLM への質問テンプレート集)
  - `instructions.md` (NotebookLM 操作マニュアル)
  - `results.md` (ユーザーが回答を貼り付けるテンプレート)
- ユーザー作業:
  1. `missing_pdfs.md` に従い不足 PDF を `pdfs/` に手動配置
  2. NotebookLM に新規ノートブック作成、`pdfs/` 全件をアップロード
  3. `notebooklm_prompts.md` のテンプレを順番に投げ、構造化回答を取得
  4. `results.md` に貼り付け、Claude Code に戻り `/integrate-factcheck`

#### CP6: 統合・修正
- 入力: レビュー結果、Factcheck 結果
- 処理:
  - NOT_SUPPORTED 判定の主張 → 修正案生成 (引用削除 or 表現緩和 or 別文献提案)
  - 複数 AI が重複指摘した箇所 → 高優先度として必ず対応
  - 致命的問題 (Claude が `[CRITICAL]` タグ付きで指摘) → 必ず対応
  - すべての修正は diff 形式で提示
- 出力: `draft_v2.md` + `revision_report.md`
- ユーザー確認: diff を承認/却下

#### CP7: Word 出力
- 入力: 承認済み `draft_v2.md` + 設定 (§8)
- 処理: python-docx で .docx 生成（テンプレートファイルが指定されていれば継承）
- 出力: `output/論文タイトル_v2.docx`

## 4. 入力仕様

### 4.1 入力形式
ユーザーは以下のいずれかのファイルをアップロードする:
- `input.md` (Markdown, 推奨)
- `input.docx` (Word)
- `input.yaml` (構造化、上級者向け)

### 4.2 必須フィールド

| フィールド | 内容 | 形式 |
|---|---|---|
| `theme` | 研究テーマ・タイトル候補 | テキスト |
| `background` | 研究背景・課題意識 | テキスト (200-800字) |
| `methods` | 用いた手法・実験条件 | テキスト + 箇条書き可 |
| `data` | 取得データの概要 (種類, 規模, 形式) | テキスト |
| `results` | 主要な結果 (図表参照可) | テキスト |
| `interpretation` | 結果の解釈・考察の方向性 | テキスト |

### 4.3 任意フィールド

| フィールド | 内容 |
|---|---|
| `citation_target_journal` | 投稿先ジャーナル名 (引用スタイル選定に使用) |
| `boost_journals` | 関連論文が多く出版される高関連性ジャーナルの複数指定 (検索ブースト用、§5.5) |
| `field` | 分野指定 (デフォルト: 理学・工学) |
| `length` | 目標文字数 (本文・要旨別) |
| `style` | 書体・フォント等の様式指定 |
| `citation_style` | 引用スタイル (IEEE / ACS / AIP / Nature / Vancouver / APA) |
| `existing_references` | 必ず引用してほしい論文リスト |
| `figures_tables` | 図表のキャプションと出現位置の指定 |

### 4.4 入力テンプレート例 (Markdown)

```markdown
# Theme
〇〇システムにおける△△手法の有効性評価

# Background
近年、…という課題が知られているが…

# Methods
- 実験装置: …
- パラメータ: …

# Data
- サンプル数: 120
- 計測項目: …

# Results
条件Aで効率が15%向上した…

# Interpretation
これは…のメカニズムによると考えられる…

# Config
citation_target_journal: IEEE Transactions on ...
boost_journals:
  - name: "IEEE Trans. on Pattern Analysis and Machine Intelligence"
    issn: "0162-8828"
  - name: "Nature Machine Intelligence"
    issn: "2522-5839"
length:
  abstract: 250
  body: 8000
citation_style: IEEE
```

## 5. 文献検索仕様

### 5.1 OpenAlex API + Exa MCP 連携

**OpenAlex API (主検索)**

OpenAlex は 2.5 億件超の学術論文を収録する完全無料の API で、タイトル・著者・DOI・ISSN・掲載誌・被引用数・OA フルテキストリンクを構造化データとして一括取得できる。

設定:
- Endpoint: `https://api.openalex.org/works`
- Polite pool: クエリに `mailto=<user_email>` を付与してレート制限を緩和 (10 req/sec)
- フィールド絞り込み: `select=id,doi,title,authorships,primary_location,publication_year,cited_by_count,open_access` で帯域を節約
- 追加登録・API キー不要

**Exa MCP (補完検索)**

OpenAlex で取得件数が閾値 (デフォルト 30 件) 未満の場合に補完として起動する。arXiv プレプリントや新興誌など OpenAlex の収録遅延をカバーする用途に限定する。

`.claude/mcp.json` に登録:
```json
{
  "mcpServers": {
    "exa": {
      "command": "npx",
      "args": ["-y", "@exa/mcp-server"],
      "env": { "EXA_API_KEY": "<key>" }
    }
  }
}
```

Exa でヒットした論文で DOI が欠損している場合は CrossRef API (§5.6) で補完する。

### 5.2 検索戦略

1. **主検索 (OpenAlex)**: テーマ + 手法のキーフレーズで `search` クエリ。`filter=publication_year:>YYYY` で期間指定 (デフォルト過去10年)
2. **補完検索 (Exa)**: OpenAlex ヒット数 < 30 件の場合、`arxiv.org`, `nature.com`, `science.org`, `ieee.org`, `mdpi.com` 等のドメインを指定して追加検索
3. **CrossRef 補完**: DOI 欠損論文は `api.crossref.org/works?query=<title>&query.author=<author>` で DOI・ISSN を取得
4. **重複排除**: DOI 一致で統合。DOI なしの場合はタイトルの Levenshtein 類似度 ≥ 0.9 で重複候補を警告

### 5.3 Q1/Q2 フィルタリング

#### データソース
[Scimago Journal Rank](https://www.scimagojr.com) の年次 CSV を `data/sjr_<year>.csv` として配置 (年1回手動更新)。

#### 判定ロジック
1. 論文のメタデータから ISSN (print / electronic 両方) を取得
2. SJR CSV と ISSN で突合 (優先)
3. ISSN 突合失敗時、誌名の正規化マッチ:
   a. 誌名を小文字・句読点除去・略語展開 (辞書: `data/journal_abbr.json`) で正規化
   b. SJR CSV の誌名も同様に正規化
   c. Levenshtein 類似度 ≥ 0.85 を一致とみなす
   d. 候補が複数ある場合は最高類似度を採用。0.85 未満はマッチ失敗扱い
4. マッチ失敗: `on_no_match` 設定に従い除外 (デフォルト) または採用
5. `bypass_sjr_for_targets: true` の場合、`boost_journals` 指定誌は SJR フィルタをバイパスして採用候補に残す

#### 略語辞書 (`data/journal_abbr.json`) 例
```json
{
  "ieee trans pattern anal mach intell": "ieee transactions on pattern analysis and machine intelligence",
  "nat mach intell": "nature machine intelligence",
  "j chem phys": "the journal of chemical physics",
  "phys rev lett": "physical review letters"
}
```
初期辞書は主要学術誌約 500 誌を収録。`scripts/add_abbr.py` で追加登録可能。

### 5.4 出力フォーマット

`references_candidates.md`:
```markdown
| ID | Authors | Year | Title | Journal | Quartile | DOI | Score | Boosted | Source |
|----|---------|------|-------|---------|----------|-----|-------|---------|--------|
| R1 | Smith et al. | 2024 | ... | Nature Comm. | Q1 | 10.1038/... | 0.92 | 0.92 | open |
| R2 | Tanaka et al. | 2023 | ... | IEEE TPAMI | Q1 | 10.1109/... | 0.88 | 1.32 | target |
| R3 | Lee et al. | 2024 | ... | Pattern Recogn. | Q1 | 10.1016/... | 0.71 | 0.71 | open |
```
- `Score`: 正規化済み関連性スコア (0–1)
- `Boosted`: boost_journals 適用後のスコア (`target_boost` 乗算済み)。ブーストなし論文は `Score` と同値
- `Source`: `open` = 通常検索ヒット、`target` = boost_journals 指定誌ヒット、`target+net` = 引用ネットワーク拡張ヒット

末尾にジャーナル別件数分布を出力:
```markdown
## Distribution
- IEEE TPAMI (target): 4 / 20
- Nature Machine Intelligence (target): 3 / 20
- Pattern Recognition: 2 / 20
- ...
- Target journal ratio: 7/20 (35%)  ← max_target_citation_ratio 以内
```

### 5.5 ターゲットジャーナル・ブースト

ユーザーが `boost_journals` を指定した場合、その分野で関連性の高い論文を高精度に取り込むためのブースト機構を有効化する。完全な絞り込みではなく、指定外の論文も並行して検索しつつ重み付けする方式とする。

#### 効果と狙い
1. **語彙・フレーミング整合**: ターゲット誌コミュニティの用語と問題設定に沿った引用候補を優先抽出
2. **査読者期待への適合**: 当該誌または近接誌の重要論文を漏らさない
3. **Q1/Q2 事前検証**: ユーザーが指定する時点で関連性が担保されているため、SJR 未収載の新興誌でも採用候補にできる (`bypass_sjr_for_targets: true`)

#### 処理パイプライン
1. **ベース検索** (OpenAlex 主検索 + Exa 補完): §5.2 の戦略で広く検索
2. **ターゲット限定検索**: 各 `boost_journals[i]` ごとに OpenAlex `filter=primary_location.source.issn:<ISSN>` で追加検索を並列実行
3. **スコア統合**:
   - ベース検索ヒットの relevance score を正規化 (0–1) してベースラインとする
   - boost_journals 掲載論文には係数 `target_boost` (デフォルト 1.5) を乗算して `Boosted` スコアを算出
4. **予約スロット適用**: 候補総数 `max_candidates` のうち `reserve_slots` 件を boost_journals に優先割当 (ただし Boosted スコアが閾値 0.3 未満の場合は埋めない)
5. **上限制御**: 最終的に boost_journals からの採用比率が `max_target_citation_ratio` を超える場合、超過分を指定外論文に置換
6. **(オプション) 引用ネットワーク拡張**: `expand_citation_network: true` のとき、boost_journals ヒット論文の被引用論文・参照論文を 1 段辿り、候補プールに追加 (Q1/Q2 フィルタは適用)

#### 副作用と対策
- **エコーチェンバー化リスク**: `max_target_citation_ratio` (デフォルト 0.4) で上限制御
- **新興誌の SJR 未収載**: `bypass_sjr_for_targets: true` でバイパス可能
- **可視化**: §5.4 のジャーナル分布出力でユーザーが採用段階で偏りを確認できる

#### 設定例 (`config.yaml` 抜粋)
```yaml
literature:
  boost_journals:
    - name: "IEEE Trans. on Pattern Analysis and Machine Intelligence"
      issn: "0162-8828"
    - name: "Nature Machine Intelligence"
      issn: "2522-5839"
  target_boost: 1.5
  reserve_slots: 5
  max_target_citation_ratio: 0.4
  bypass_sjr_for_targets: true
  expand_citation_network: false
```

### 5.6 CrossRef API による BibTeX/CSL-JSON 取得

`scripts/bib_fetcher.py` が `references_candidates.md` の DOI リストを入力として BibTeX を生成する。

- **Endpoint**: `https://api.crossref.org/works/{DOI}`
- **出力形式**: CSL-JSON → BibTeX 変換 (`citeproc-py` または `bibtexparser` を使用)
- **エラーハンドリング**:
  - 取得失敗 (DOI 不正・404・レート制限超過) は `missing_bibtex.md` に記録して次の DOI に進む
  - レート制限: `X-Rate-Limit-Limit` / `X-Rate-Limit-Interval` ヘッダーを読み取り、超過時は指数バックオフ (1s → 2s → 4s、最大 3 回リトライ)
- **出力**: `checkpoints/cp1_references.bib`
- **Polite pool**: クエリに `mailto=<user_email>` を付与して安定アクセス (無料)


1. **自動取得 (OA優先)**: OpenAlex の `open_access.oa_url` フィールドを優先取得。次に arXiv, PubMed Central, DOAJ のリンク
2. **ユーザー手動**: 取得失敗した論文を `missing_pdfs.md` に列挙し、ユーザーが所属機関等から入手して `pdfs/` に配置

```

- Verdict: SUPPORTED / PARTIAL / NOT_SUPPORTED
- Evidence: 該当箇所を該当論文から直接引用 (なければ「該当箇所なし」)
- Comment: 補足 (1-2文)
```

- `instructions.md` にノートブック分割手順を明記

## 7. マルチ AI レビュー仕様

### 7.1 レビュー観点と AI 割当

| 観点 | AI | 方式 | プロンプト要旨 |
|---|---|---|---|
| 論理構成・展開の妥当性 | Claude (reviewer Subagent) | **自動** | IMRaD 構成と主張の流れを批判的に評価 |
| 文体・読みやすさ・冗長性 | Claude (reviewer Subagent) | **自動** | 学術文体としての改善案 |
| 学術的正確性・先行研究の網羅性 | ChatGPT (with search) | Human-in-the-loop | 重要文献の欠落を指摘、概念定義の正確性 |
| 統計・データ解釈の妥当性 | Gemini | Human-in-the-loop | 数値表現、統計記述の問題点を検出 |

Claude レビューは `/review-packet` 実行と同時に reviewer Subagent が自動起動し、`results/claude.md` を生成する。ユーザーは ChatGPT・Gemini の 2 件のみ手動で実行すればよい。

※ 各 AI への担当割当は `config.yaml` の `review.assignments` で変更可能。

### 7.2 レビューパケット構造

各プロンプトファイル (`prompts/<ai_name>.md`) は以下を含む:
1. ロール指定 (「あなたは○○分野の査読者である」)
2. 評価観点 (上表の該当観点を詳細化)
3. 入力 (`draft_v1.md` の全文またはセクション切り出し)
4. **構造化出力指定**
   ```
   以下の形式で出力せよ:
   - 全体評価 (1-5)
   - 主要な指摘 (箇条書き、各「セクション名 / 指摘内容 / 修正提案」)
   - 致命的な問題 (あれば [CRITICAL] タグ付き)
   - 良い点 (1-3件)
   ```

### 7.3 統合ロジック (CP6)
- 複数 AI が同じ箇所を指摘 → 高優先度
- `[CRITICAL]` タグ付き問題 → 必ず対応
- 改善提案は diff 形式でユーザーに提示し承認制

## 10. Claude Code 設定詳細

### 10.1 Slash Commands (例: `/draft`)

`.claude/commands/draft.md`:
```markdown
---
description: アウトラインに従ってドラフトを執筆する
---

Writer Subagent を呼び出し、`checkpoints/cp2_outline.md` を入力として
セクション別に本文を執筆せよ。

- 引用は `references_candidates.md` に記載された ID のみを `[REF: <ID>]` 形式で挿入すること。リスト外の ID を生成してはならない
- 執筆完了後、出力テキスト中の全 `[REF: *]` を references_candidates.md の ID 列と照合し、存在しない ID があれば当該セクションを再生成すること (最大 2 回リトライ)
- Abstract (config.yaml の abstract_chars に従う) と Keywords (5〜8 語) を生成すること
- 各セクションの文字数目安を遵守
- 専門用語は最初の出現で英語併記
- セクション完了ごとに progress.yaml を更新すること
- 出力は `checkpoints/cp3_draft_v1.md`
```

### 10.2 Subagents

#### Writer
役割: アウトライン + 研究要素 + 引用候補 → 本文セクション執筆 + Abstract + Keywords
プロンプト方針:
- 学術文体 (である調)
- 一文 40-60 字を目安
- 主張 → 根拠 → 引用 の論理構造を保持
- 推測と事実を明確に区別
- 引用は `references_candidates.md` の ID 列の値のみ使用（制約厳守）

#### Reviewer
役割: `draft_v1.md` → 論理構成・文体レビュー (`results/claude.md`)
プロンプト方針:
- IMRaD 各セクションの論理的一貫性を評価
- 致命的問題は `[CRITICAL]` タグ付きで明記
- 指摘は「セクション名 / 指摘内容 / 修正提案」形式で出力

#### Integrator
役割: レビュー/factcheck 結果を本文に統合
プロンプト方針:
- NOT_SUPPORTED 判定 → 引用削除 or 別文献提案 or 主張緩和
- 複数 AI の重複指摘 → 必ず対応
- 修正は diff で提示、ユーザー承認後に反映

### 10.3 Skills

#### `sjr-filter`
- Python スクリプト経由で `data/sjr_<year>.csv` と `data/journal_abbr.json` を読込
- ISSN → Quartile 判定、ISSN 失敗時は正規化誌名 + Levenshtein で突合
- 関数: `filter_q1_q2(references: list) -> list`

#### `citation`
- `cp1_references.bib` (BibTeX) 入力 → 指定スタイルの参考文献リスト出力
- IEEE, ACS, AIP, Nature, Vancouver, APA 対応

#### `docx-export`
- 本文 Markdown + config → .docx
- `export.template` 指定時はテンプレートファイルのスタイルを継承

### 10.4 MCP

| Server | 用途 | コスト |
|---|---|---|
| Exa | 補完文献検索 | 無料枠あり (補完用途のみのため枠内に収まる想定) |
| (任意) Google Drive | 出力共有 | 無料 |

### 10.5 チェックポイント状態管理・エラー回復

各プロジェクトの `checkpoints/progress.yaml` で進捗を管理する:

```yaml
current_cp: 3
cp3_sections_done:
  - introduction
  - methods
cp3_sections_pending:
  - results
  - discussion
  - conclusion
  - abstract
  - keywords
last_updated: "2026-05-29T14:32:00"
error_log: []
```

- 各スラッシュコマンドは起動時に `progress.yaml` を確認し、未完了の箇所から再開する
- セクション・ステップ完了のたびに即時更新（クラッシュ時の工程損失を最小化）
- エラー発生時はエラー内容を `error_log` に追記してから処理を停止
- ユーザーは `progress.yaml` を直接編集して特定の CP に巻き戻すことができる

## 13. 制約と前提

### 13.1 NotebookLM
- 公式 API なし → 完全自動化不可
- Free 版のソース数上限あり → 大量引用論文の場合は分割運用
- 利用規約上、商用大量利用は不可

### 13.2 Web 版 AI の利用規約
各 AI の Web 版を業務的・大量に使用する場合、利用規約 (rate limit, 商用利用条件) の確認をユーザー責任とする。

### 13.3 PDF 入手
非 OA 論文はユーザーの所属機関アクセス前提。これがない場合、Q1/Q2 縛りと OA 縛りの両立で引用候補が減少する。

### 13.4 SJR データ
年次更新。最新 Quartile 反映には CSV の手動更新が必要。

### 13.5 Q1/Q2 の定義
SJR ベース。JCR (Journal Citation Reports, 有料) の Quartile とは一致しないことに注意。

### 13.6 OpenAlex API
- **レート制限**: anonymous 利用は 10 req/10sec。`mailto=<email>` を付与した polite pool では制限が緩和される。大量検索時は polite pool を必ず使用すること
- **収録遅延**: 直近 1〜2 か月の論文は未収録の場合がある。最新プレプリントは Exa 補完検索でカバーする
- **フィールド正確性**: メタデータの ISSN や誌名に欠損がある論文が一定割合存在する。CrossRef API での補完 (§5.6) で対処する
