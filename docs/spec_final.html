Version: 1.2  |  Date: 2026-06-02  |  Target field (Phase 1): 理学・工学  |  Repository: https://github.com/dan23514/paper-agent

1. 概要

1.1 目的

ユーザーから提供される研究要素（テーマ・手法・データ・結果・解釈）を入力として、Q1/Q2 ジャーナル掲載論文のみを引用した日本語の学術論文ドラフトを生成し、ファクトチェック・マルチAIレビューを経て Word (.docx) 形式で出力する半自動エージェントを構築する。

1.2 設計原則

#原則内容
1Claude Code 単一環境完結追加フレームワーク（LangGraph 等）を導入せず、Claude Code のスラッシュコマンド・Subagent・Skill・MCP で構成する
2Human-in-the-loop によるコスト圧縮ChatGPT・Gemini のレビューはユーザーが Web 版を手動で呼び出すリレー方式とし、API 従量課金を回避する。Claude レビューは Claude Code 内で自動実行する
3引用ファクトチェックの独立化汎用 LLM の引用捏造リスクを排除するため、ファクトチェックは NotebookLM に分離する
4チェックポイント駆動全自動ではなく、明示的なチェックポイントごとに成果物を出力し、ユーザーが介入・承認してから次工程に進む
5引用捏造防止verify_refs.py が本文中の全 [REF: Rx] を候補リスト ID と照合。無効 ID は即時リジェクト・再生成
6DOI 検証優先文献は必ず CrossRef API で DOI・タイトル・年を実在確認してから BibTeX を採用する

1.3 想定コスト

  • Claude Code Pro または Max サブスクリプション
  • OpenAlex API：完全無料（登録不要）
  • CrossRef API：完全無料
  • Exa MCP：補完用途のみのため無料枠内に収まる想定
  • NotebookLM / ChatGPT (web) / Gemini (web)：いずれも無料枠を活用

4. 文献検索仕様

4.1 OpenAlex API（主検索）

  • Endpoint: https://api.openalex.org/works
  • Polite pool: クエリに mailto=<user_email> を付与してレート制限を緩和（10 req/sec）
  • フィールド絞り込み: select= で取得フィールドを限定し帯域を節約
  • 収録論文数: 2.5 億件以上（arXiv・PubMed Central・DOAJ 等を包含）

4.2 Q1/Q2 フィルタリング

Scimago Journal Rank (SJR) の年次 CSV（data/sjr_YYYY.csv）を使用。

優先順位マッチング方法閾値
1位ISSN（完全一致）—
2位誌名の正規化 + Levenshtein 類似度≥ 0.85
特例boost_journals 指定誌は SJR バイパス可（bypass_sjr_for_targets: true）—

4.3 ターゲットジャーナル・ブースト

boost_journals を指定すると、指定誌の論文に係数（target_boost、デフォルト 1.5）を乗算して優先度を上げる。上限制御（max_target_citation_ratio、デフォルト 0.4）でエコーチェンバー化を防ぐ。

5. Factcheck 仕様

5.1 Factcheck Packet 構成

factcheck_packet/
├── claims.md              本文中の全引用主張（主張 + 引用ID + DOI のテーブル）
├── pdfs/                  自動取得した OA-PDF（PDF マジックバイト検証済み）
├── missing_pdfs.md        ユーザーが手動入手すべき PDF リスト
├── notebooklm_prompts.md  NotebookLM への検証プロンプト集
├── instructions.md        NotebookLM 操作マニュアル
└── results.md             結果記入テンプレート（空）

5.2 NotebookLM の判定基準

Verdict意味/integrate-factcheck での処理
SUPPORTED主張は論文に明確に支持されている変更なし
PARTIAL部分的に支持されている表現を緩和する修正案を提示
NOT_SUPPORTED論文に該当箇所がない引用削除・表現変更・代替文献提案

6. マルチ AI レビュー仕様

観点AI方式
論理構成・文体Claude (reviewer Subagent)自動実行
学術的正確性・先行研究の網羅性ChatGPT (with search)Human-in-the-loop
統計・データ解釈の妥当性GeminiHuman-in-the-loop

6.1 統合ロジック

  • [CRITICAL] タグ付きの指摘 → 必ず対応
  • 2 つ以上の AI が同じ箇所を指摘 → 高優先度として必ず対応
  • 改善提案は diff 形式でユーザーに提示し承認制

8. エラー回復・状態管理

各プロジェクトの checkpoints/progress.yaml で進捗を管理する。

  • 各スラッシュコマンドは起動時に progress.yaml を確認し、未完了の箇所から再開する
  • セクション・ステップ完了のたびに即時更新（クラッシュ時の工程損失を最小化）
  • エラー発生時は error_log に追記してから処理を停止
  • ユーザーは progress.yaml の current_cp を手動編集して特定の CP に巻き戻せる
