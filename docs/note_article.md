# Claude Code で学術論文の執筆を半自動化してみた話

[画像: ヘッダー画像（例: Claude Code のターミナル画面 + 論文原稿のコラージュ）]

## はじめに

研究者として一番大変な作業のひとつが、論文を書くことです。

実験やデータ解析に比べると、執筆は「やればできる」はずなのに、なぜか手が止まります。参考文献を探して、読んで、引用する場所を決めて、文章を組み立てて……気づけば数週間が過ぎていた、という経験をされている方も多いのではないでしょうか。

私も今ちょうど、CMIP6（第6期結合モデル相互比較計画）のマルチモデルデータを使って日本海通過流の将来変化を解析した論文を書き始めようとしていました。データ解析は終わっているのに、執筆だけがずっと先送りになっている状態です。

「**AI を使えば、この執筆プロセスを効率化できないか？**」

そう思い立って、Claude Code で論文執筆エージェントを作ることにしました。

---

## 作ったもの

**paper-agent** という名前のシステムです。

やっていることをひとことで言うと、「**Q1/Q2 ジャーナルに掲載された論文だけを引用した学術論文のドラフトを、自動生成して Word ファイルで出力する**」エージェントです。

[画像: ワークフロー全体図（CP0〜CP7 の流れ、矢印でつないだフロー図）]

大まかな流れはこうです。

1. **CP0**: 研究テーマ・手法・結果などを書いた入力ファイルを読み込む
2. **CP1**: OpenAlex API で関連論文を検索し、Scimago のデータで Q1/Q2 だけに絞り込む
3. **CP2**: IMRaD 構成のアウトラインを生成する
4. **CP3**: セクションごとに本文・Abstract・Keywords を執筆する
5. **CP4**: Claude が自動レビュー、ChatGPT・Gemini に手動で投げてレビュー結果を統合する
6. **CP5**: 引用の裏付けを NotebookLM で検証する（Factcheck）
7. **CP7**: Word (.docx) に出力する

すべて Claude Code のスラッシュコマンドで操作でき、「/load」「/search」「/draft」……と順番に実行していくだけです。

---

## 技術的なポイント

### 引用の捏造をどう防ぐか

大規模言語モデルが論文執筆で最も怖いのが、存在しない論文を引用する「ハルシネーション」です。

このシステムでは、**引用 ID の検証機構** を設けています。まず CP1 の文献検索で収集した候補（R1, R2, ...）だけを「使ってよい引用 ID リスト」として確定させます。そして執筆後に `verify_refs.py` というスクリプトが本文中の全 `[REF: Rx]` を候補リストと照合し、リスト外の ID が1件でも含まれていたらそのセクションを再生成します。

CP1 の文献収集段階でも、すべての論文を CrossRef API で DOI・タイトル・年を実在確認してから BibTeX を取得しています。

[画像: 引用ガード機構のイメージ図（候補リスト → 検証 → 合格/リジェクトのフロー）]

### なぜ OpenAlex を主軸にしたか

最初は Exa MCP（Web 検索エンジン）を主検索エンジンにしていましたが、学術論文の構造化メタデータ（DOI・ISSN・著者・掲載誌名）を安定して返せないことがわかりました。OpenAlex は 2.5 億件以上の論文を収録した無料の学術 API で、DOI・ISSN・OA リンクまで整ったデータが取得できます。Exa は OpenAlex でカバーできないプレプリントや新興誌の補完にのみ使うよう設計を変更しました。

### Human-in-the-loop という割り切り

完全自動化を目指すと、どうしても ChatGPT・Gemini の API 課金が必要になります。今回は意図的に「**Claude Code の中でできることは自動化し、それ以外は人間が手動で行う**」という設計にしました。

ChatGPT と Gemini のレビューはユーザーが Web 版に手動で貼り付けて回答を保存するだけ。Factcheck は NotebookLM に PDF をアップロードして行います。どれも無料枠で完結します。

## 実際に動かしてみた

今回のターゲット論文は、日本海通過流の将来変化を CMIP6 の 17 モデルで解析したものです。テーマ・手法・結果を書いた `input.md` を用意して、CP0 から順番にコマンドを実行していきました。

[画像: ターミナルのスクリーンショット（/search を実行して文献候補リストが出力されているところ）]

### CP1 で起きた小さなドラマ

文献検索の段階で、ChatGPT がレビュー時に「基幹文献が不足している」と指摘してきた文献群を、CP1 の段階で CrossRef で検証して追加しました。

このとき、自動解決スクリプトが 6 件を誤マッチさせていました（例: Ohshima 1994 を検索したら 2017 年の別の論文が返ってきた）。全件を CrossRef で題目・著者・年・掲載誌を確認してから採用したことで、捏造リスクを早い段階で排除できました。

### Factcheck で引用ミスを発見

CP5 の Factcheck 段階で、CP1 の採番修正（不要論文の削除・再採番）によって引用 ID がずれていた箇所を2件発見しました。

- 「通過流の**年周期**」→ 渦循環の論文（Park 2015）を引用していた → 正しい年周期の論文（Kida 2015）に修正
- 「海峡の**地形・摩擦**」→ 適切な Form Drag の論文に修正

これはまさにファクトチェックの工程が本来果たすべき役割で、自動化されたパイプラインでも人間によるレビューが必要な理由がここにあると実感しました。

[画像: NotebookLM での Factcheck 作業のスクリーンショット]

## 結果

最終的に、入力 → Word 出力まで **1〜2日** で通すことができました。

生成されたドラフトは約 12,100 字（目標 14,000 字の 86%）。残りは私が追記する予定ですが、論文の骨格・流れ・引用は完成しています。

| | 内容 |
|---|---|
| 引用候補 | 29件（DOI・BibTeX 全件検証済み） |
| 引用スタイル | APA（著者-年方式） |
| Factcheck 結果 | 21件引用中 16件 SUPPORTED、1件 PARTIAL 対応済み |
| 出力フォーマット | Word (.docx)、游明朝・Times New Roman、A4・行間1.5 |

## 公開しています

このシステムは GitHub で公開しています。

**https://github.com/dan23514/paper-agent**

セットアップに必要なものは Python 3.10+、Claude Code、そして SJR の CSV（無料ダウンロード）だけです。

```
pip install -r requirements.txt
# → data/sjr_2025.csv を配置
# → scripts/config.example.py を config.py にコピーしてメールアドレスを設定
# → /load <project_name> から開始
```

## おわりに

論文執筆エージェントを作って気づいたのは、「**AI が最も役立つのは骨格を作ることで、肉付けは人間の専門知識が必要**」ということです。

引用の検証・レビューへの応答・図番号の対応・追加統計解析……人間が判断しなければならない部分は思ったより多く残ります。それでも「白紙から始める」という最もつらい部分を乗り越えてくれるだけで、執筆の心理的障壁はかなり下がりました。

理学・工学の論文を書いている方、ぜひ試してみてください。フィードバックもお待ちしています。
